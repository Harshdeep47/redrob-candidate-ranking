# Redrob Candidate Discovery & Ranking — Senior AI Engineer (Founding Team)

An AI system that ranks 100,000 candidate profiles against a single job
description the way a thoughtful recruiter would: by reading career history,
weighing verified signal over self-reported keywords, and explicitly
penalizing the keyword-stuffing trap the JD itself warns about.

## Quick start

```bash
pip install -r requirements.txt
python rank.py --candidates path/to/candidates.jsonl --out submission.csv
```

That's the single command that reproduces the submission CSV from a
candidates file.

**Two timing numbers, because precomputation is real and the spec explicitly
allows it to sit outside the 5-minute budget** (see "Two-stage design"
below):

- **First run** (no cached `artifacts/`, or a `--candidates` file that
  doesn't match whatever is cached): the command above also builds the
  cached features + TF-IDF index, which takes ~45-80s for the full 100K
  pool. Total observed wall-clock for this first run: **~45-60s**.
- **The actual time-boxed ranking step** (artifacts already cached from a
  previous run against the *same* candidates file): **~4-5 seconds,
  <1GB RAM**, CPU-only, zero network calls. This is the part the spec's
  5-minute/16GB budget is meant to apply to.

Either way you are nowhere near the 5-minute ceiling. To run precompute and
ranking as two explicit, separately-timed steps:

```bash
python -m src.precompute --candidates path/to/candidates.jsonl --out artifacts
python rank.py --candidates path/to/candidates.jsonl --artifacts artifacts --out submission.csv
```

## Sandbox / demo

`app.py` is a small Streamlit app satisfying the hackathon's mandatory
sandbox requirement: upload a `candidates.jsonl` (≤100 candidates recommended,
two samples are provided in `data/`), click **Run ranking**, and it runs the
exact same `rank.py`/`precompute.py` pipeline used for the full submission,
in-process, then lets you preview and download the resulting CSV.

Run locally:

```bash
pip install -r requirements.txt streamlit
streamlit run app.py
```

To deploy (satisfies the spec's hosted-sandbox requirement): push this repo
to GitHub, then create a new app at [streamlit.io/cloud](https://streamlit.io/cloud)
pointing at `app.py` in this repo. No secrets or API keys needed — everything
is local CPU compute. Once deployed, put the live URL into
`submission_metadata.yaml` under `reproducibility.sandbox_link`.

## Architecture

### Two-stage design: precompute vs. rank

The hackathon brief constrains the *ranking* step to ≤5 minutes, ≤16GB RAM,
CPU-only, no network/API calls — but doesn't time-box getting there. So the
pipeline is split:

- **`src/precompute.py`** (not time-boxed): reads `candidates.jsonl` once,
  extracts structured features for every candidate, fits a TF-IDF vectorizer
  over the full corpus + the JD text, and caches everything to `artifacts/`
  as parquet/npz files. Takes ~45-80s for 100K candidates on a single core.
- **`rank.py`** (the actual time-boxed deliverable): loads the cached
  artifacts, computes a hybrid score, applies gating penalties, and writes
  the top-100 CSV. Takes ~4-5 seconds.

If `artifacts/` is missing, `rank.py` runs precompute automatically first, so
the single-command promise above is honest even though precompute is
logically a separate stage.

Artifacts are **not** checked into git, on purpose: they're built from a
specific `candidates.jsonl`, and `precompute.py` stamps a fingerprint
(file size + hash of the first/last 1MB) of the source file it was built
from into `artifacts/source.meta.json`. `rank.py` checks this fingerprint
before trusting cached artifacts and silently regenerates them if the
`--candidates` file given doesn't match — so it's never possible to
accidentally rank against stale data left over from a different input file
(e.g. Redrob's own held-out evaluation file vs. the 100K file used during
development).

### Why TF-IDF instead of a neural embedding model

I initially planned to use `sentence-transformers` for semantic similarity.
I dropped it after actually trying to install it here: it pulls a multi-GB
CUDA/PyTorch stack even when you only want CPU inference, which is a real
risk for a "must reproduce cleanly in a sandboxed Docker container" grading
step — exactly the kind of dependency bloat that breaks on a clean rebuild.

`TfidfVectorizer` + cosine similarity (scikit-learn, already a core
dependency) does the job here because the domain vocabulary is narrow and
mostly closed (133 known skills, ~48 known titles, one JD) — this isn't
open-domain semantic search, it's matching specific technical phrases
("vector search", "hybrid retrieval", "production deployment") against a
single fixed JD. It's also fully auditable: every similarity score traces
back to literal shared terms, which is much easier to defend in a live
interview than "the neural net said so."

### The hybrid score

```
base_score = 0.28 * semantic_similarity        (TF-IDF cosine vs JD)
           + 0.22 * core_skill_score            (taxonomy-weighted, verified > self-reported)
           + 0.16 * production_signal           (shipped-to-production language, title-gated)
           + 0.10 * experience_fit              (JD's 6-8yr ideal band)
           + 0.12 * availability                (recency, response rate, open-to-work)
           + 0.07 * location_fit
           + 0.05 * notice_fit

final_score = base_score
            x title_relevance_gate              (0.05 - 1.0, see below)
            x consulting_only_penalty
            x architecture_only_penalty
            x title_chaser_penalty
            x cv_speech_only_penalty
            x visa_gate_penalty
            x (0.01 if honeypot else 1.0)
```

Penalties are **multiplicative**, not additive, deliberately — they encode
things the JD frames as disqualifying-or-near-disqualifying conditions, not
softly-weighted preferences. A 1% honeypot multiplier or a 5% title-relevance
gate means no amount of keyword overlap elsewhere can drag an obviously
wrong profile into the top 100.

### The central design problem: title vs. description trustworthiness

The most important thing I found in this dataset (`notebooks/explore3.py`):
`career_history[].description` text is topically matched to its own
`career_history[].title` ~100% of the time when the title is ML/AI-relevant,
but only ~16% of the time for common non-technical titles (HR Manager, Sales
Executive, etc.) — the other 84% of the time the description is drawn from
an unrelated, shuffled pool.

That finding directly shapes `title_relevance_gate()` in `src/features.py`:
**title is the dominant, trustworthy signal; free text can't be used to
grant relevance to an otherwise-irrelevant title.** This is also why
`production_signal_score()` only reads descriptions attached to a
plausibly-relevant role — reading all descriptions indiscriminately would
let a content writer's randomly-assigned "deployed to production" sentence
count as evidence for an HR Manager.

This was found the hard way: see `git log` — v1 of the pipeline ranked an
HR Manager at Wipro into the top 100 on the strength of self-reported
"Recommendation Systems" and "Sentence Transformers" skills. Fixing that is
the single biggest jump in output quality in this repo's history.

### Honeypot / impossible-profile detection

Two clean, low-noise signals (no false positives found on manual review):

1. A skill marked `expert` proficiency with `duration_months == 0`.
2. `years_of_experience` differing from `sum(career_history duration_months)/12`
   by more than 3 years.

68 candidates trip one of these (zero overlap between the two signals),
which is close to the brief's mention of "~80" planted honeypots. We don't
hard-exclude them — we suppress their score by 99%, which has the same
practical effect without needing to claim we found every single one.

## Known limitations / what I'd improve with more time

- **JD parsing is manual.** There's only one JD for this challenge, so I read
  it by hand rather than building an LLM-based parser. A production version
  would replace `src/jd_requirements.py` with an automated parse step so the
  system generalizes to new JDs without a code change.
- **TF-IDF won't generalize to JDs with different vocabulary** the way a
  neural embedding model would. For this single-JD challenge that's a
  reasonable trade against compute-constraint robustness; for a real product
  I'd want sentence-transformers running as a precompute-time-only dependency
  (still no GPU needed at rank time, since embeddings would be cached) with
  a fallback to TF-IDF if the dependency isn't available.
- **Reasoning generation is template-based, not LLM-generated**, intentionally,
  to guarantee zero hallucination risk under a no-network ranking constraint.
  A v2 could use an LLM *offline at precompute time* to polish phrasing while
  still constraining it to the same fact-slots, getting more natural language
  without reintroducing hallucination risk into the time-boxed step.
- **Weights were chosen by reasoning about the JD's stated priorities**, not
  fit against the hidden ground truth (we don't have it). With labeled data
  this would be a logistic regression / LightGBM ranker instead of a manual
  weighted sum.
